1. Key Principle: Traceability to BRs

Every test must link back to a specific behavioural requirement (BR1–BR9) from our requirements document. The marker wants to see that our tests verify what we said the system would do. Always reference the BR number in the Test Oracle field.

Our requirements document defines 9 behavioural requirements (BR1–BR9) with Gherkin scenarios. The test report must demonstrate that we tested against these specifications. Where the implementation differs from the original BR (e.g., BR2.1 uses local reposition instead of scene loading), document this honestly.

2. Section 1: What Has Been Tested (20%)

For each test type, list the modules/functions and state the oracle (where the expected result comes from). The oracle should reference our BRs or Gherkin scenarios.

Unit Testing — Items Under Test

  • QuizManager.SubmitAnswer(int selectedIndex) — Oracle: BR3.2/BR3.3 states correct answer advances quiz, incorrect triggers retry (up to 3 attempts)
  • QuizManager.LoadQuiz(string worldId) — Oracle: BR3.1 requires quiz to present questions with four options; JSON must parse correctly from StreamingAssets/Data/Worlds/{worldId}.json
  • ScoreLog.RecordScore(string worldId, int score, int totalQuestions) — Oracle: BR5.2 states completing a world updates progress; returns true only when score exceeds stored best
  • WrongAnswerLog.LogWrongAnswer(string worldId, QuestionData q) — Oracle: design specification requires wrong answers to be logged for analytics
  • InventoryManager.AddItem() / PopItem() / HasItem() — Oracle: BR4.1 states keys/tools are added to player’s inventory
  • Cauldron.CheckRecipe() — Oracle: design specification requires all 7 correct ingredients for win; any wrong ingredient triggers failure with adaptive hint (BR8.1)
  • GameState.currentState setter — Oracle: design specification requires state change events to fire for Hoverboard/Station transitions (supports BR2.1)
  • RobotHintAI JSON response parsing — Oracle: BR8.1/BR8.2 requires AI Orb to provide hints; LLM responses must parse into {thought, hint, emotion} fields

System Testing — Items Under Test

  • Full quiz flow from entry to completion — Oracle: BR3.1–BR3.4 acceptance criteria
  • Station/portal transition with fade effect — Oracle: BR2.1 acceptance criteria
  • AI hint delivery after incorrect quiz answer — Oracle: BR8.1 acceptance criteria
  • Robot escort collects item and delivers to cauldron — Oracle: BR9.2 acceptance criteria
  • Score persistence across sessions — Oracle: BR5.1/BR5.2 acceptance criteria
  • Cauldron puzzle: collect, submit, validate — Oracle: design specification derived from BR4.1

User Acceptance Testing (UAT) — Items Under Test

  • BR3 User Story: “As a player, I want to complete quizzes…” — Oracle: end-user confirms quiz is completable and score is shown
  • BR8 User Story: “As a player, I want to receive hints when I get stuck…” — Oracle: end-user confirms hints are helpful without giving away answers
  • BR9 User Story: “As a player, I want the AI Orb to guide me…” — Oracle: end-user confirms robot provides guidance
  • BR2 User Story: “As a player, I want to travel to different skill worlds…” — Oracle: end-user confirms portal transitions work
  • BR1 User Story: “As a player, I want to understand where I am…” — Oracle: end-user confirms spawn shows all portals

3. Section 2: Test Cases (50%)

Important notes:
  • Our BR status section (Section 2.1 of the tech report) already noted some requirements are “Partially Met” or “Not Met” — tests should reflect this honestly.
  • Reference the specific BR number in each Test Oracle field.
  • We don’t have a formal test framework (no NUnit) — our tests are manual tests in Unity Play Mode. That is acceptable; just be explicit about it.
  • If a test fails, record it, note the severity, fix the bug, and run Cycle 2.

5. Section 4: Summary and Reflection on Test Quality (10%)

Template — fill in after running all tests:

What to write:

1. Results summary: “We executed X unit tests, Y system tests, and Z UAT tests. Of these, A passed and B failed.”

2. Equivalence classes used:

  • Quiz answers: correct index vs incorrect indices (UT-02, UT-03)
  • Score comparison: new score > best, new score ≤ best, first score for world (UT-04, UT-05)
  • Inventory state: empty vs non-empty (UT-06)
  • Cauldron ingredients: all correct vs any wrong vs too few (UT-07)
  • JSON input: valid file vs missing file vs malformed file (UT-01)

3. Coverage gaps (be honest):

  • No automated unit tests — all testing was manual
  • LLM responses are non-deterministic, so AI hint quality cannot be deterministically verified
  • VR-specific interactions (grabbing, throwing) are difficult to automate and were tested manually on-device
  • Some BRs marked “Partially Met” in Section 2.1 (e.g., BR2.1 uses local reposition, not scene load) — tests reflect actual behaviour
  • Multiplayer (BR5.6) and advanced AI emotional modelling (BR9.3) are “Will Not Have” and were not tested

4. Critical failures and resolution: List any High severity failures found during testing and whether they were fixed (Cycle 2) or remain known issues.

5. Lessons learned: “In future, we would integrate Unity Test Framework earlier to enable automated regression testing, particularly for the quiz scoring and inventory logic.”
